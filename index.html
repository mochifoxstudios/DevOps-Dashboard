Receiving Webhooks

Last updated · April 30, 2026 · est. 6 min read

Webhooks let your application be notified about events that happen in the gateway. Configure an endpoint and subscribe to one or more event types. Each delivery is signed; verify the signature before trusting the payload.

Quick start

  • Create an endpoint at POST /webhooks/payments
  • Subscribe to payment_intent.succeeded
  • Verify X-Signature with your whsec_… secret
  • Respond with 2xx within 5 seconds

Example payload

{ "id": "evt_3PqL2X…", "type": "payment_intent.succeeded", "created": 1715774538, "data": { "object": { "amount": 2499, "currency": "usd" } } }

Retries

Failed deliveries retry with exponential backoff for up to 72 hours. Idempotency keys are derived from the event id.

Guides
Working with Context-Snap snapshots
Concept · capture & restore reproducible local state
guide

A snapshot is a frozen record of everything that defines the behavior of your local services at one moment: environment variables, active process tree, branch and commit, open ports, and optionally a working-tree diff. Snapshots are append-only and stored under ~/.devops-local/snapshots/.

Capture a snapshot

  1. Open Context-Snap (⌘ 1).
  2. Give the snapshot a descriptive name — e.g. preflight-2026-05-15-checkout.
  3. Pick what to include. Env vars, PIDs, and git are always recommended; open ports and working-tree diff are larger but invaluable when reproducing intermittent bugs.
  4. Click Snapshot now, or press ⌘⇧S from anywhere in the app.
Secrets are auto-redacted. Any value that matches an API-key or token heuristic is replaced with "sk_test_•••…". Add custom patterns under Settings → Local Security.

Restore a snapshot

Click Restore snapshot in the view header and pick a snapshot. Restoration is partial by default — it sets env vars and recommends commands to bring processes back, but never kills your existing processes without confirmation.

Share with a teammate

Use Export to write a self-contained .snap archive (encrypted with your passphrase). Drop the file into someone else's Context-Snap to import.

Caching documentation for offline reading
Doc Scraper · pin versioned references locally
guide

The Doc Scraper fetches a documentation page, normalizes it to Markdown, and pins a version-tagged copy under ~/.devops-local/docs/. After it's cached, you can browse and search the page entirely offline.

Cache a page

  1. Open Quick-Docs Scraper (⌘2).
  2. Paste a URL into the address bar. The scraper supports https://, file://, and a GitHub mode for repo READMEs.
  3. Click Fetch Document. The page is normalized, code blocks preserved, and the result indexed for search.

Air-gap mode

Air-gap is on by default. The scraper can only reach hosts on your allowlist (Settings → Network Isolation). Add a domain there before first fetch.

Cache lifecycle

  • Cached docs are tagged with the date and the source's ETag/Last-Modified when available.
  • The default TTL is 90 days. After that, the entry stays usable but is shown as stale with a one-click re-fetch.
  • When your disk budget is reached, oldest stale docs are pruned first.
Auditing dependencies offline
Dependency Map · find outdated and vulnerable packages
guide

The Dependency Map parses a project manifest and the matching lockfile, then cross-references each pinned version against a local CVE cache. All work happens on disk — no registry calls.

Supported manifests

  • package.json + package-lock.json / pnpm-lock.yaml / yarn.lock
  • requirements.txt and pip-tools output
  • Cargo.toml + Cargo.lock
  • go.mod + go.sum
  • Gemfile.lock

Refresh the CVE cache

Run devops-local sync cves from your terminal, or pop open Settings → Network Isolation and use Sync CVE database while the allowlist is open. The cache file lives at ~/.devops-local/cves.db.

Reading the table

  • up to date — pinned version matches the latest you have indexed.
  • outdated / major behind — a newer release exists in your local index.
  • CVE-XXXX-YYYY — flagged vulnerability; click the badge for severity, fixed versions, and references.
Tailing and filtering log files
Log-Tail · plain text, glob, and regex filters
guide

Log-Tail keeps a streaming connection to any file the OS lets you read. Filters apply to live lines and to the scrollback buffer.

Filter syntax

  • plain text — case-insensitive substring match.
  • "quoted phrase" — exact-phrase match.
  • /regex/ — JavaScript regex; add i, m, s after the closing slash for flags.
  • error AND timeout, error OR warn, NOT debug — boolean operators.
  • level:error, req:b3c1d — key:value match on parsed JSON or logfmt fields.
// pin every Stripe webhook signature mismatch /stripe-webhook.*signature mismatch/i // errors that mention pool exhaustion, not warnings level:error AND "pool exhausted"

Pinned errors

Lines matching your error patterns (Settings → Watchers & Triggers) are pinned to a sidebar in the Log-Tail view. Pins survive restarts and can be exported alongside snapshots.

Filing a reproducible bug report
Issue Filler · structured Markdown your team can act on
guide

The Issue Filler is a wizard that turns the context you've already captured into a complete Markdown report. Every step links the active snapshot, the pinned logs, and any flagged CVEs — so the resulting issue is reproducible by default.

  1. Triage — title, severity, component.
  2. Environment — auto-filled from the active snapshot. You can swap in any other snapshot at any time.
  3. Reproduction — the steps you took; the answer to "what should have happened instead?"
  4. Logs & artifacts — attach pinned log ranges and the working-tree diff.
  5. Review & export — copy to clipboard, save as .md, or open in your default issue tracker.
Wizard state is auto-saved every keystroke under ~/.devops-local/drafts/. Close the app mid-report; come back to the same step tomorrow.
Understanding the autonomous agent
Background service · watchers, schedules, and limits
concept

The autonomous agent is a long-running local process that performs the work the UI describes — capturing snapshots, parsing manifests, watching log files, applying retention policy. The window in front of you is just one view onto it.

Lifecycle

  • Starts at boot (toggleable in Settings → Autonomous Agent → Launch at boot).
  • Runs as the current user; never as root.
  • Listens on 127.0.0.1:7822 for the UI. The port never accepts non-loopback connections.
  • Honors your CPU/memory ceilings; the OS kills and restarts the agent if it exceeds them.

What it does without you

  • Auto-snapshots on every git checkout across branches.
  • Re-parses manifests when their files change on disk.
  • Tails configured log paths and pins lines matching your error patterns.
  • Prunes old snapshots once your disk budget is reached.
  • Runs the scheduled audit (default 03:00 daily).
Pausing the agent stops all autonomous behavior but keeps the app window usable for browsing existing data. Use Settings → Autonomous Agent → Pause.
Workflows
Pre-flight before a staging deploy
5 minutes · proves your local state matches what's about to ship
workflow
  1. Pull the release branch and start your services normally.
  2. Open Context-Snap, name the snapshot preflight-<sha>, include env + PIDs + git + diff, capture.
  3. Open Dependency Map, re-scan, and confirm zero red CVE badges.
  4. Open Log-Tail with the filter level:error OR level:warn and watch for ~60 seconds of nominal traffic.
  5. Run your smoke tests; if a step fails, jump straight to Issue Filler — the snapshot is already linked.
Tip: bind this workflow to a single devops-local preflight CLI alias — see Settings → Local Toolchain.
Diagnose a flaky integration test
15 minutes · capture a passing and failing run, diff them
workflow
  1. Run the test in a loop. The first time it passes, capture a snapshot named flaky-pass.
  2. When it fails, capture flaky-fail immediately — env, PIDs, ports, and diff.
  3. In Context-Snap, click Compare with the two snapshots selected. Anything orange has drifted between runs.
  4. Open Log-Tail filtered by the test's request id (it usually leaves a req_… in the logs) and pin every error line.
  5. File the issue from the failing snapshot. Attach the pinned log ranges.
Reproduce a customer-reported bug
20 minutes · use the customer's reported context to rebuild local state
workflow
  1. Grab the customer's request id, version string, and the time window from the support ticket.
  2. Check out the matching git SHA. Capture a snapshot named repro-<ticket>.
  3. Open Log-Tail, filter by the request id and the time window. Pin the error chain.
  4. Run Dep Map to confirm your local dependencies match the version string the customer reported.
  5. Open Issue Filler and pick the Customer repro template. Snapshot and pins auto-link.
Be careful with PII. Default redaction handles tokens and keys, but customer emails and IDs do not match generic patterns. Add a custom redaction rule before pasting customer logs.

Frequently Asked
Does anything I capture leave my machine?
No. In Offline Local Mode, every byte is written to ~/.devops-local. The only outbound traffic is the doc scraper, and only to hosts on your allowlist.
How are secrets handled in snapshots?
Auto-redacted via a built-in pattern set: API keys, JWTs, basic-auth headers, common connection strings. Add custom regexes under Settings → Local Security.
Can I share a snapshot with a teammate?
Yes. Export to .snap (encrypted with your passphrase). They drop the file into their own workspace and unlock it with their copy of the passphrase.
Which manifest formats are supported?
npm, pnpm, yarn, pip, cargo, go modules, and bundler. Composer (PHP) and Maven (Java) are on the roadmap — vote in the issue tracker.
Does the agent run when I'm signed out?
Only if Launch at boot is enabled. Otherwise it starts when you sign in and stops when you sign out.
What if I forget my passphrase?
There is no recovery service. Encrypted snapshots become unreadable; everything else (unencrypted docs, settings) is still usable. Rotate the key from Settings → Local Security after recovery.
How big can ~/.devops-local get?
Capped at your disk budget (default 2 GB). Snapshots are diff-compressed and the doc cache deduplicates resources.
Troubleshooting
The agent won't start
Sidebar shows "Local agent unavailable" or the pulse dot is red
issue
  1. Check the agent log at ~/.devops-local/agent.log. The last error line will almost always tell you why it bailed.
  2. Common cause: another process already bound 127.0.0.1:7822. Find the offender with lsof -i :7822 and either stop it or change the port under Settings → Network Isolation.
  3. If the agent is being killed by the OS for memory, raise the ceiling under Settings → Resource Limits.
  4. As a last resort, run devops-local doctor from a terminal — it prints a diagnostic and offers safe-mode startup.
Storage usage keeps climbing
Disk budget warning, or auto-prune doesn't seem to run
issue
  1. Open Settings → Storage & Retention. Confirm the disk budget is set and that retention isn't "Keep all".
  2. Run Compact now under that section. This rewrites snapshots as diffs against their predecessor.
  3. Working-tree diffs are the biggest single contributor. Toggle them off in Context-Snap if you don't need them for every capture.
  4. If usage is still high, check ~/.devops-local/docs/ — a single page can pull in MBs of images and assets. Set the doc cache TTL lower.
A watcher stopped firing
Auto-snapshots or auto-scans no longer happen on change
issue
  1. macOS, Linux, and Windows each cap the number of file-watch descriptors per process. Heavy node_modules trees can hit the cap.
  2. On macOS, raise the cap with sudo sysctl -w kern.maxfiles=1048576 and kern.maxfilesperproc=524288.
  3. On Linux: echo fs.inotify.max_user_watches=524288 | sudo tee -a /etc/sysctl.conf then sudo sysctl -p.
  4. Add high-churn paths (build output, caches) to the watcher ignore list under Settings → Watchers & Triggers.
Doc scraper returns an empty page
The fetch succeeds but the cached page is blank or broken
issue
  1. The site likely renders content with client-side JS. Enable JS evaluation for that domain under Settings → Network Isolation — the scraper boots a headless renderer just for allowlisted hosts.
  2. Some pages refuse non-browser user agents. Set a per-host UA override.
  3. If the cache file exists but is empty, delete ~/.devops-local/docs/<host>/<slug>.md and re-fetch — a partial fetch can poison the cache.
Locked out: forgot the passphrase
Snapshots are encrypted and won't open
issue

By design, DevOps Local has no recovery service. Your passphrase is the only key. If you've lost it:

  1. Settings → everything else still works. Cached docs and pinned issues are unencrypted.
  2. To start fresh: Settings → Local Security → Rotate local key with a new passphrase, then erase encrypted snapshots from Storage & Retention.
  3. Recover the passphrase from your password manager, or have a teammate share a snapshot that they can re-encrypt for you.
Glossary
snapshot
An immutable record of your local state at one moment — env, processes, branch, optionally a diff.
watcher
A long-running file or process observer maintained by the agent. Fires when its target changes.
manifest
The declared-dependencies file for a project: package.json, Cargo.toml, go.mod, etc.
lockfile
The resolved, exact-version companion to a manifest. Where the Dependency Map gets its truth.
pinned error
A log line that matched an error pattern and was set aside for review. Survives restarts.
air-gap mode
Default network posture. The agent refuses every outbound connection except scraper requests to allowlisted hosts.
allowlist
Domains the doc scraper is permitted to reach while air-gap is on.
retention
The rule that decides which snapshots and cached docs are kept vs. pruned. Configured in Storage & Retention.
workspace
A named project root. Each workspace has its own watchers, snapshots, and pinned issues.
leader key
The modifier used for tool-switching shortcuts. ⌘ on macOS by default; remappable.
diff-aware snapshot
A snapshot stored only as the difference from its predecessor. Saves ~40% of disk on average.
stale doc
A cached document past its TTL. Still readable; flagged for re-fetch.
